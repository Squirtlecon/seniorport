Dungeon Crawler

A basic dungeon crawler game with procedural generation. Enemies are not yet implemented, as it is a work in progress. Originally built for a hackathon. Built using the Godot Engine in GDScript.

Background

During March of 2024, I participated in an school hackathon with some friends. We had decided towards doing a dungeon crawler with random generation, in order to give high replayability. We intended to add various enemies, items and npcs.

The main goal of the hackathon was in order to get procedural generation working. We were given a basic item and player, so we built off that. Due to previous experience in Godot, we decided to use it to code.


The Team

It started off with Isaac Zhang and I working together. Later on Ellis Park joined us with much larger ambitions, which led to the addition of Eric Homer. Isaac had been assigned towards coding the procedural generation. Ellis had been assigned to creating tiles for the dungeons. Eric was assigned to working on music and enemies.








My Role

I was sent to be the primary sprite artist, and create items. I created the basic character and spritesheet, with over 40 animation sprites. I then created the alternative character, and made multiple enemy sprites. Lastly I worked on the UI and items.

Requirements & Needs

The hackathon required us to build a game, presenting the options of top down or side scroller. We decided on top down, and got to drafting In order to fit the summer theme, we intended to have multiple dungeons, each in a island or "rest area" of sorts.



The Initial Design

We started off with the starter repository, then started off on the procedural generation. We originally got it worked with a map of sorts , but not rooms. Due to time restraints the generation only included one type of room, but that is a work in progress.

The next step was to add life to the dungeon, with enemies and items. While Eric worked on coding enemies, I created sprites for him and coded the UI.

Unfortuntely, we encountered many obstucles on our journey. The first issue started off as the guide we used was in a different version of godot, which significant changes. Issues persisted, but were eventually solved. The next major issue involved Eric's enemy. The enemy's code did not work on anyone else's computer, even though we used the same version.

We decided to upload the work and go home. We had decided to continue our work there. However, Isaac's computer had an issue uploading, but luckily he was able to sort it out.

Thankfully, at our houses the coding went smoothly. We finished the procedural generation, and created a weapon.

The UI

While creating the UI, we had to think of what to put on it. We decided on having the stats of health, stamina, and mana. Health would be used to take damage, stamina for sprinting, and mana for spells. I created multiple iterations, before settling on three final designs (seen in the picture). Here we chose the one best suited towards the style of the dungeon.



The Final Design

For the final design, we kept some of the features from the initial design - such as the navigation bar, most of the navigation labels, and the simple subscription form. We also kept the grid layout to display past newsletters.

We did change the color scheme to a combination of purples and blues, which better matched our other brand and social media materials. We added an additional page for events and moved the subscription form to the home page.

The website was built in HTML, CSS, and JavaScript. The form posts the submitted emails to a google sheets, using a custom API built using the platform SheetDb. The website is deployed through Vercel.
